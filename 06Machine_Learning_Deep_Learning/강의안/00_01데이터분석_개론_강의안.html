PART 0 데이터 분석 개요

0부. 데이터 분석이란 무엇인가

데이터 분석(Data Analysis)은 관측된 데이터를 기반으로 **패턴을 발견하고, 설명하고, 예측하거나, 의사결정을 돕는 과정**이다. 기술(도구/모델)보다 중요한 것은 항상 **“무엇을 알고 싶은가?”라는 질문(Problem Definition)**이다.

설명(Descriptive) 무슨 일이 일어났는가? - KPI/대시보드, 리포팅 - 탐색적 분석(EDA)

EDABIDashboard

예측(Predictive) 앞으로 무엇이 일어날 것인가? - 분류/회귀/시계열 예측 - 위험/수요/이탈 확률

ClassificationRegressionForecasting

처방(Prescriptive) 그래서 무엇을 해야 하는가? - 최적화/실험(A/B) - 정책/추천/가격 전략

OptimizationExperimentDecision

제품화(Productization) 분석 결과를 “서비스”로 만들기 - 모델 운영(MLOps) - 모니터링/드리프트/재학습

MLOpsMonitoringDrift
Data수집·정제
Analysis통계·ML·DL
Insight원인·패턴
Decision정책·전략
데이터 분석의 끝은 “리포트”가 아니라 “의사결정(행동)”이다.

핀테크에서의 활용(0부 관점)

  • 설명: 결제 성공률, 앱 전환율, 부정거래(의심) 비율, 고객 민원 유형을 대시보드로 상시 모니터링
  • 예측: 대출 연체 가능성, 고객 이탈 가능성, 카드 사용액/예금 유입(시계열) 예측
  • 처방: 금리/한도/프로모션을 최적화하거나, A/B 테스트로 UI·알림 정책 의사결정
  • 제품화: “신용평가/사기탐지/추천”을 API로 운영하고 드리프트 감지 후 재학습
핵심은 “모델”보다, 문제정의→데이터→운영까지 이어지는 전체 흐름이다.
PART 1 분석 유형

1부. 데이터 분석의 종류

정답이 있는가? (지도/비지도) 예측 대상이 숫자인가/범주인가? (회귀/분류) 시간 축이 중요한가? (시계열) 이 3가지로 대부분의 문제를 분류할 수 있다.

분류 분석 (Classification) 목표: 데이터를 정해진 범주 중 하나로 구분 산출: 확률(0~1) + 임계값 정책 예시: 스팸/정상, 이탈/유지, 질병 유무

Precision/RecallROC-AUCThreshold

회귀 분석 (Regression) 목표: 연속적인 수치를 예측 산출: 예측값 + 오차(불확실성) 예시: 가격, 매출, 시간, 점수 예측

MAE/RMSEResidualUncertainty

군집 분석 (Clustering) 목표: 정답 라벨 없이 유사한 데이터 묶기 산출: 군집 ID + 군집 해석(프로파일링) 예시: 고객 세그먼트, 행동 유형

K-MeansDBSCANSilhouette

차원 축소 (Dimensionality Reduction) 목표: 고차원 정보를 낮은 차원으로 압축 산출: 임베딩/좌표(시각화, 노이즈 제거) 예시: 2D 시각화, 특징 압축, 검색 기반

PCAt-SNE/UMAPEmbedding

연관 규칙 (Association) 목표: 함께 발생하는 패턴 찾기 예시: 장바구니 분석(상품 A→B 동시구매)

AprioriSupport/Confidence

이상탐지 (Anomaly Detection) 목표: 정상과 다른 패턴(희귀 사건) 탐지 예시: 센서 이상, 사기/부정, 품질 이상

Isolation ForestAutoEncoderRule+Model

핀테크에서의 활용(1부: “문제 유형” 매핑)

  • 분류: 대출 승인/거절, 연체 위험(고/중/저), 부정거래 의심(Yes/No), 고객 이탈(Yes/No)
  • 회귀: 적정 한도/금리 산출, 예상 소비액/예금 잔액, 수수료/수익 예측
  • 군집: 고객을 “투자성향/소비패턴/대출성향” 세그먼트로 나눠 개인화 마케팅
  • 차원축소: 거래/로그의 고차원 특징을 임베딩으로 압축해 검색·탐지·시각화에 활용
  • 연관규칙: “이 상품을 산 사람은 이런 금융상품도 가입” → 번들/교차판매 후보
  • 이상탐지: AML(자금세탁) 의심 패턴, 계정 탈취 징후, 비정상 결제/로그인 탐지
PART 2 Machine Learning Tabular

2부. 머신러닝 기반 데이터 분석

머신러닝은 “특징(Feature)”을 바탕으로 예측 규칙을 학습한다. 표형(Tabular) 데이터에서는 여전히 **트리 기반 모델(Gradient Boosting)**이 강력한 베이스라인이다.

선형 모델 - 해석이 쉬움 - 빠르고 안정적 예: 선형/로지스틱 회귀

트리/앙상블 - 비선형 패턴 강함 - 피처 엔지니어링과 궁합 예: RandomForest, XGBoost

커널/거리 기반 - 데이터 형태에 민감 - 스케일링 중요 예: SVM, KNN

실무 체크리스트 (표형 데이터) - 데이터 누수(Leakage) 점검: 미래 정보가 피처에 섞이지 않게 - 불균형(Imbalance): Precision/Recall 중심으로 평가 + 임계값 최적화 - 검증 전략: 시간 데이터는 TimeSplit / 그룹은 GroupKFold - 설명가능성: 중요 피처/기여도(SHAP 등)로 의사결정 근거 제공

문제정의타깃/단위/기간
피처집계·전처리
학습CV/튜닝
배포서빙/모니터링
머신러닝 프로젝트는 “모델”이 아니라 “파이프라인”이 성패를 좌우한다.

핀테크에서의 활용(2부: 표형 ML이 강한 영역)

  • 신용/리스크 스코어링: 고객의 소득·부채·거래패턴·연체이력 등 표형 특징으로 연체확률(PD) 예측
  • 사기탐지(FDS): 결제/이체의 속도·빈도·디바이스·지역·시간대 특징으로 이상 거래 분류
  • 개인화 추천: 고객-상품 상호작용(클릭/가입/해지) 기반으로 적합 상품 추천(기본은 트리/랭킹 모델부터)
  • 운영 최적화: 콜센터/심사 인력 수요 예측, 연체 회수 우선순위 산정(처방/우선순위 모델)
  • 설명가능성: 금융은 “왜 거절/승인인가?”가 중요 → SHAP 등으로 피처 근거를 함께 제공
PART 3 Deep Learning Representation

3부. 딥러닝 기반 데이터 분석

딥러닝은 “피처를 사람이 설계”하기보다, 데이터에서 **표현(Representation)**을 직접 학습한다. 대량 데이터/복잡한 신호(이미지·텍스트·음성)에서 특히 강하다.

핵심 개념 - 손실함수(Loss): 무엇을 잘 맞추게 할 것인가 - 최적화(Optimizer): 어떻게 학습할 것인가 - 일반화(Regularization): 과적합을 어떻게 막을 것인가

LossOptimizerRegularization

대표 구조 - MLP: 표형/기본 - CNN: 이미지/공간 패턴 - RNN/LSTM: 순서 데이터(최근은 Transformer로 대체)

MLPCNNTransformer

딥러닝 실무 포인트 - 데이터 라벨 품질이 성능의 상한선을 결정 - 증강(Augmentation)과 정규화가 중요 - 학습/검증 분리(누수 방지) + Early Stopping - 배포 시 지연시간/비용 고려(경량화: distillation/quantization)

핀테크에서의 활용(3부: DL이 특히 유리한 데이터)

  • 텍스트: 민원/상담 로그 분류(불만 유형, 긴급도), 공시/뉴스 감성·이슈 추출(주의: 투자 권유와는 별개)
  • 시계열: 거래 시퀀스 기반 계정탈취/이상 징후 탐지(“행동의 연속성”을 학습)
  • 음성: 콜센터 STT 후 “고객 감정/불만/해지 의도” 탐지 및 상담 품질 평가
  • 경량화: 모바일 앱/엣지 환경에서 동작시키려면 distillation/quantization이 실제 비용 절감 포인트
PART 4 LLM RAG

4부. LLM 기반 데이터 분석

LLM은 텍스트를 “이해”한다기보다, 대규모 텍스트에서 학습한 패턴으로 **다음 토큰을 예측**한다. 데이터 분석에서는 LLM을 ‘예측기’뿐 아니라 **문서 이해·요약·분류·질의응답·코딩 보조**에 활용한다.

LLM이 잘하는 것 - 자연어 요약/정리/분류 - 규칙이 복잡한 텍스트 라벨링 보조 - SQL/파이썬 초안 생성 및 디버깅 힌트

SummarizationNL2SQLLabel Assist

LLM의 위험 - 할루시네이션(근거 없는 생성) - 최신성/사실성 보장 어려움 - 개인정보/보안 이슈

HallucinationPrivacySafety

RAG(검색 증강 생성) 패턴 LLM이 “기억”으로 답하게 하지 말고, 먼저 검색으로 근거를 가져온 뒤 그 근거만으로 답하게 만든다. - (1) 문서 청킹/임베딩 → (2) 유사도 검색 → (3) 근거 컨텍스트로 생성 - 결과: 사실성↑, 출처 제시 가능

Ingest문서 수집
Embed벡터화
Retrieve검색
Generate근거 기반 답변
LLM을 “근거 기반”으로 운용할 때, 데이터 분석/업무 자동화에 안전하게 쓸 수 있다.

핀테크에서의 활용(4부: LLM/RAG가 강한 업무)

  • 규정/약관 Q&A: 내부 규정·상품설명서·FAQ를 RAG로 연결해 “근거 있는 답변” 제공
  • 고객 상담 자동화: 상담 시나리오/스크립트 추천, 문의 분류, 상담 요약(단, 개인정보 마스킹/접근통제 필수)
  • 문서 처리: 심사 서류(설명서/증빙) 핵심 항목 추출 → 업무 처리 시간 단축
  • 데이터 분석 보조: NL2SQL로 탐색 쿼리 초안 생성, 지표 정의 문장화, 리포트 초안 작성
  • 가드레일: “출처 없는 생성 금지”, “민감정보 차단”, “금융 조언/투자 권유 제한” 같은 정책을 함께 설계
PART 5 Vision Multimodal

5부. Vision / 멀티모달 데이터 분석

Vision 분석은 이미지/영상에서 패턴을 찾아 **분류·탐지·분할·추적**을 수행한다. 최근에는 텍스트+이미지를 함께 다루는 **멀티모달(vision-language)**이 빠르게 확산 중이다.

대표 과제 - Image Classification: 이미지 분류 - Object Detection: 물체 위치/종류 - Segmentation: 픽셀 단위 분할 - OCR: 글자 인식 및 구조화

ClassificationDetectionSegmentationOCR

실무 포인트 - 라벨링 비용이 큼 → 액티브 러닝/반자동 라벨링 - 데이터 편향/도메인 차이(조명·각도) 대응 - 배포 비용(서빙/GPU) 고려한 경량화

LabelingBiasDeployment

멀티모달(LLM+Vision) 활용 예 - 이미지 설명/요약(리포트 자동화) - 문서 이미지(PDF 스캔)에서 표/키값 추출 - “이미지 + 질문” 형태의 QA(지원/검수 자동화)

핀테크에서의 활용(5부: Vision/멀티모달)

  • KYC/서류 자동화: 신분증/증빙 서류 OCR → 항목 추출(이름/번호/발급기관) + 진위/품질 체크(흐림/가림)
  • 영수증/세금계산서: 경비처리·소득공제 입력 자동화(분류 + 금액/가맹점/일자 추출)
  • 문서 QA: 스캔 PDF에서 표/조항을 찾아 “이 항목이 어디에 있나요?” 같은 질의응답
  • 부정 방지: 계정 생성 시 제출 이미지(프로필/서류) 중복·위변조 탐지(단, 규제·인권 이슈 고려 필요)
PART 6 직군 커리어 로드맵

6부. 데이터 분석 직군과 커리어 로드맵

데이터 관련 직무는 모두 “데이터를 다룬다”는 공통점이 있지만, 어디에 더 집중하는가(분석 · 모델 · 시스템 · 운영)에 따라 역할이 명확히 나뉜다. 약자가 아니라 실제 업무 관점에서 이해하는 것이 중요하다.

데이터 분석가 (Data Analyst) - 비즈니스 문제를 데이터로 해석하고 지표로 설명 - KPI 정의, 대시보드, 정기/비정기 리포트 작성 - 실험(A/B Test) 결과 분석 및 의사결정 지원 → “무슨 일이 일어났는가, 왜 그런가”를 설명하는 역할

SQL 통계 EDA BI 도구

데이터 사이언티스트 (Data Scientist) - 예측/최적화 문제를 수학·통계·머신러닝으로 해결 - 분류·회귀·추천·이상탐지 모델 설계 - 피처 엔지니어링 및 모델 성능 평가·해석 → “앞으로 무엇이 일어날지, 어떤 선택이 좋은지”를 모델로 답하는 역할

머신러닝 모델링 피처 엔지니어링 평가지표

머신러닝 엔지니어 (Machine Learning Engineer) - 학습된 모델을 실제 서비스에서 사용 가능하도록 구현 - 모델 API 개발, 배포, 성능/지연시간 관리 - 모델 모니터링 및 재학습 파이프라인 구축 → “모델을 코드와 시스템으로 안정적으로 운영”하는 역할

핀테크 취업 관점 체크포인트(6부)

  • 분석가: “지표 정의→원인 분석→실험 설계”를 말로 설명할 수 있어야 함(퍼널/코호트/AB)
  • DS: “평가/임계값/비용”을 함께 설계(예: 사기탐지에서 FP 비용이 너무 크면 운영이 망가짐)
  • DE: 거래 데이터는 품질/정합성이 핵심(중복, 지연 도착, 재처리, 감사 로그)
  • MLE/LLM: 개인정보·보안·접근통제·모니터링까지 포함한 “운영 설계” 경험이 강점
핀테크는 “정확도”만큼이나 규제/보안/운영을 함께 보는 사람이 채용에 강합니다.
WRAP-UP 정리

정리. 한 장으로 보는 데이터 분석 & 핀테크 적용

분석 관점 - 설명: KPI/리포트/원인 - 예측: 위험/이탈/수요 - 처방: 최적화/A-B - 제품화: 운영/모니터링

핀테크 적용 관점 - 리스크: 신용/연체/부정 - 개인화: 추천/세그먼트 - 자동화: 서류/상담/규정 QA - 운영: 드리프트/재학습/감사

실무 한 줄 결론

  • 핀테크 데이터 분석은 “정확한 문제정의 + 데이터 품질 + 운영 설계”가 성능을 이깁니다.
  • 모델은 도구일 뿐, 서비스는 정책(임계값/룰) + 모니터링 + 재처리까지 포함입니다.
